**Security Warning**: This endpoint is unauthenticated and exposes potentially sensitive webhook processing data. Only deploy on trusted networks (VPN, internal network). See [Log Viewer Security](/security/log-viewer-security) for deployment guidance.
</Warning>

## Endpoint

```
GET /logs/api/workflow-steps/{hook_id}
```

Retrieve detailed timeline and execution data for individual workflow steps within a webhook processing flow. This endpoint provides granular, step-by-step analysis of webhook processing workflows with detailed timing, execution status, and diagnostic information.

## Parameters

<ParamField path="hook_id" type="string" required>
  GitHub webhook delivery ID (X-GitHub-Delivery header value)
  
  Example: `f4b3c2d1-a9b8-4c5d-9e8f-1a2b3c4d5e6f`
</ParamField>

## Response

<ResponseField name="hook_id" type="string">
  The webhook delivery ID requested
</ResponseField>

<ResponseField name="workflow_metadata" type="object">
  <Expandable title="properties">
    <ResponseField name="repository" type="string">
      Repository name (org/repo format)
    </ResponseField>
    <ResponseField name="event_type" type="string">
      GitHub event type (pull_request, push, etc.)
    </ResponseField>
    <ResponseField name="initiated_at" type="string">
      Workflow start timestamp (ISO 8601)
    </ResponseField>
    <ResponseField name="total_duration_ms" type="number">
      Total workflow duration in milliseconds
    </ResponseField>
    <ResponseField name="total_steps" type="number">
      Total number of workflow steps
    </ResponseField>
    <ResponseField name="steps_completed" type="number">
      Number of successfully completed steps
    </ResponseField>
    <ResponseField name="steps_failed" type="number">
      Number of failed steps
    </ResponseField>
    <ResponseField name="steps_skipped" type="number">
      Number of skipped steps
    </ResponseField>
  </Expandable>
</ResponseField>

<ResponseField name="execution_timeline" type="array">
  Array of workflow step executions in chronological order
  
  <Expandable title="step properties">
    <ResponseField name="step_id" type="string">
      Unique step identifier
    </ResponseField>
    <ResponseField name="step_name" type="string">
      Human-readable step name
    </ResponseField>
    <ResponseField name="sequence" type="number">
      Step execution order
    </ResponseField>
    <ResponseField name="started_at" type="string">
      Step start timestamp (ISO 8601)
    </ResponseField>
    <ResponseField name="completed_at" type="string">
      Step completion timestamp (ISO 8601)
    </ResponseField>
    <ResponseField name="duration_ms" type="number">
      Step duration in milliseconds
    </ResponseField>
    <ResponseField name="status" type="string">
      Execution status: `completed`, `failed`, or `skipped`
    </ResponseField>
    <ResponseField name="operation_type" type="string">
      Type of operation: `security`, `data_processing`, `api_call`, etc.
    </ResponseField>
    <ResponseField name="details" type="object">
      Step-specific execution details and metadata
    </ResponseField>
    <ResponseField name="error" type="object" optional>
      Error information if step failed
      <Expandable title="error properties">
        <ResponseField name="type" type="string">
          Error type/exception name
        </ResponseField>
        <ResponseField name="message" type="string">
          Error message
        </ResponseField>
        <ResponseField name="traceback" type="string">
          Full error traceback
        </ResponseField>
      </Expandable>
    </ResponseField>
  </Expandable>
</ResponseField>

## Use Cases

1. **Debug specific workflow step failures** - Detailed error information for failed steps
2. **Analyze step-by-step performance** - Identify slow operations and bottlenecks
3. **Monitor operation success rates** - Track failure patterns across workflows
4. **Generate audit trails** - Compliance and monitoring with detailed execution logs
5. **Identify resource bottlenecks** - Find operations consuming excessive resources
6. **Track GitHub API usage** - Monitor API rate limiting per workflow step
7. **Investigate integration issues** - Debug external service integration problems

## Error Responses

<ResponseField name="404" type="object">
  Hook ID not found in logs
  
  ```json
  {
    "detail": "Workflow steps not found for hook_id: f4b3c2d1..."
  }
  ```
</ResponseField>

<ResponseField name="500" type="object">
  Internal server error
  
  ```json
  {
    "detail": "Error retrieving workflow steps: <error message>"
  }
  ```
</ResponseField>
